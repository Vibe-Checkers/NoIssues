Why didn't these repos succeed on the first attempt?

-----

Issue: First attempt used `npm install` without `--ignore-scripts`. Husky git hooks
ran during install and failed because git binary was not in the Docker image.
Why not immediate: The agent's preparation phase didn't analyze package.json for
`husky` as a devDependency, nor did it check CI workflow commands (which use npm ci).
Fix applied in attempt 2: Added `git` to apt-get install, used `npm ci`.

-----
django/django — 2 attempts (language mislabeled as Node.js)
Issue: First attempt generated a Node.js Dockerfile for a Python project.
Why not immediate: Language detection returned "Node.js" (probably found
package.json from docs/tests tooling). The wrong base image and build commands
were used entirely.
Fix applied in attempt 2: Agent discovered Python files and regenerated correctly.
Note: This is entirely a language detection failure. The actual Python Dockerfile
needed no iteration.

-----
distcc/distcc — 2 attempts

SECTION 6: AGENT ARCHITECTURE NOTES (for context)
================================================================================

WORKFLOW LOOP (workflow.py):
  - Phase 1: preparation.py — language detection, CI analysis, guideline generation
  - Phase 2: up to 3 attempts, each with:
      - Build goal prompt (with lessons from previous attempts prepended)
      - LangChain ReAct agent execution
      - Post-hoc lesson extraction using LLM (from VerifyBuild + SearchDockerError outputs)
      - Enforcement: VerifyBuild must be called AFTER last WriteToFile and must succeed

TOOLS AVAILABLE TO AGENT:
  - ListDirectory, ReadFile, FindFiles, GrepFiles, WriteToFile
  - VerifyBuild: actual docker build + smoke test + LLM error analysis
  - SearchDockerError: web search + LLM to diagnose Docker errors
  - DockerImageSearch: query Docker Hub for valid image tags
  - SearchWeb: general web search

KEY ARCHITECTURAL OBSERVATIONS:
  1. DockerImageSearch exists but is NEVER called proactively. It is only called
     when the agent happens to think of it. Agents consistently pick invalid image
     tags and waste an attempt discovering this.

  2. The verification enforcement rule ("VerifyBuild must be LAST") is correct
     in principle but triggers ~9 failures because the agent writes to the file
     after verifying — perhaps because the agent is trying to "clean up" or add
     a comment. There is no prevention mechanism, only detection.

  3. Language detection is fragile for multi-language repos or repos where the
     primary language isn't the dominant file type (e.g., TensorFlow has .go files
     from its build tooling).

  4. Preparation phase does NOT read package.json engines, requirements.txt,
     pom.xml, or gradle.properties. These contain the version pins and special
     requirements that cause most 2-attempt failures.

  5. SearchDockerError is reactive only — it's called after VerifyBuild fails.
     It could be called during ANALYZE phase to proactively discover the right
     base image, build flags, and common pitfalls for the detected language.

  6. No cross-repo learning. Each repo starts fresh. Lessons from folly's
     libgoogle-glog-dev fix are not available when the next C++ repo runs.

  7. The 3-attempt limit means that repos with N sequential-dependency issues
     (where issue K is only discoverable after K-1 is fixed) have a hard ceiling.
     Spring Framework required exactly 3 ordered Gradle fixes — it would have
     needed attempt 4 to verify the final fix.

SECTION 7: DESIGN IMPROVEMENT RECOMMENDATIONS
================================================================================

Ranked by estimated accuracy improvement vs. speed/cost overhead.

TIER 1 — HIGH IMPACT, LOW COST (implement first)
-------------------------------------------------

1. PREVENT MODIFY-AFTER-VERIFY (fixes ~3 sure failures, saves ~6 wasted attempts)
   Problem: Agent writes to Dockerfile after VerifyBuild passes, breaking state.
   Fix: After VerifyBuild returns status="success", set an immutable lock flag.
        If agent calls WriteToFile after this point, intercept the call, return
        the current run as SUCCESSFUL with the locked Dockerfile, and exit.
   Cost: ~5 lines of code in workflow.py. Zero extra LLM calls.
   Repos fixed: mybatis-3, TypeScript (all 3 attempts were ordering violations),
                imgui (2/3 attempts were ordering violations).

2. PROACTIVE BASE IMAGE VALIDATION IN ANALYZE PHASE
   Problem: Agent picks non-existent Docker tags (e.g., maven:3.8.7-openjdk-17-slim),
            wastes entire attempt discovering this.
   Fix: In the system prompt / goal prompt, add explicit instruction:
        "Before writing the Dockerfile, call DockerImageSearch to verify the exact
        base image tag exists. Do not proceed without confirming."
   Cost: 1 DockerImageSearch call per repo (~1–2s). Zero extra LLM tokens beyond
         the tool call.
   Repos fixed: guava (2 attempts on bad tags), Activiti (partially).

3. RICHER PREPARATION: ANALYZE BUILD MANIFESTS
   Problem: preparation.py detects language but doesn't extract version pins,
            special deps, or build commands from manifest files.
   Fix: Add analyze_build_manifests() step in preparation.py that reads:
        - package.json → engines.node, scripts.postinstall, husky presence
        - requirements.txt / pyproject.toml → pinned deps, build-system requirements
        - build.gradle / gradle.properties → Gradle wrapper version, Java toolchain
        - setup.py / setup.cfg → ext_modules presence (indicates C extensions)
        - meson.build / CMakeLists.txt → top-level dependencies
        Inject this as structured context into the agent's goal prompt.
   Cost: ~5–10 file reads (fast). One extra LLM call to summarize. ~500–1000 tokens.
   Repos helped: numpy (missed gfortran), bootstrap (missed husky), django (language
                 mislabel), scipy (complex deps), react/react-native (pre/postinstall).

4. FIX LANGUAGE DETECTION FALLBACK
   Problem: 12 repos detected as "Unknown" — 9 of these failed. When Unknown,
            no language guidelines are injected.
   Fix: Improve detect_project_language() with secondary strategies:
        a. Check for build files explicitly: pom.xml/build.gradle → Java,
           CMakeLists.txt → C/C++, Cargo.toml → Rust, go.mod → Go.
        b. If still ambiguous, check primary file extensions but weight by
           non-test, non-docs, non-config files only.
        c. If still Unknown, have agent call SearchWeb("what language is [repo]?").
   Cost: Small additional file check. No LLM calls required for most cases.
   Repos helped: folly, json-c, imgui, msgpack-c, webview (all C/C++),
                 guava, Activiti, mybatis-3, spring-framework, RxJava (all Java).

TIER 2 — MEDIUM IMPACT, SMALL COST
------------------------------------

5. STRUCTURED CI/CD EXTRACTION
   Problem: preparation.py extracts CI text but not structured commands.
   Fix: Parse .github/workflows YAML to extract: exact build command, test command,
        package manager, Node/Python/Java version, timeout setting.
        Inject as concrete key-value pairs (not free text) into goal prompt.
   Cost: YAML parsing, no extra LLM calls.
   Repos helped: react (CI shows full COPY then install pattern), bootstrap (CI
                 shows npm ci --ignore-scripts), spring-framework (CI shows
                 Gradle flags).

6. FAILURE CLASSIFICATION BEFORE LESSON INJECTION
   Problem: After VerifyBuild fails, the agent calls SearchDockerError generically.
            For certain error types, a different tool or strategy is better.
   Fix: Classify error type from VerifyBuild output:
        - "manifest unknown" / "not found" → IMAGE_NOT_FOUND → call DockerImageSearch
        - "E: Unable to locate package X" → APT_NOT_FOUND → query apt-cache alternatives
        - "rat:check" / "license" → COMPLIANCE → log as unfixable, skip remaining attempts
        - "timed out" → TIMEOUT → increase timeout_seconds, simplify build
        - "unknown instruction" / "syntax error" → SYNTAX → lint Dockerfile first
   Cost: One regex/string match pass over error output. No extra LLM calls.
   Repos saved: commons-csv, flink (detect compliance failure early → save 2 attempts)

7. RESET DOCKERFILE STATE BETWEEN ATTEMPTS
   Problem: When attempt N fails, attempt N+1 modifies the broken Dockerfile from N,
            causing additive corruption.
   Fix: Before each retry, delete the existing Dockerfile (or save it aside).
        The next attempt starts from scratch, but with lessons in the prompt.
        This prevents the agent from "fixing" a partially-broken state.
   Cost: One file delete per retry. Zero extra LLM calls.
   Repos helped: folly (oscillating fixes), json-c (corrupt Dockerfile introduced
                 by agent), imgui (stray characters from previous attempt).

8. INCREASE/ADAPT TIMEOUT BY PROJECT COMPLEXITY HEURISTIC
   Problem: React and Flink exceed the 600s VerifyBuild timeout.
   Fix: Detect complexity signals before building:
        - Java multi-module (many pom.xml or build.gradle files) → 1200s timeout
        - C++ with many CMake targets → 900s timeout
        - Large repos (>500MB) → 900s timeout
        Default remains 600s.
   Cost: File count check. No extra LLM calls.
   Repos saved: react, flink (partially — flink also has RAT issue).

TIER 3 — LOWER PRIORITY (cross-run improvements)
--------------------------------------------------

9. CROSS-REPO PATTERN CACHE
   Problem: No learning across repos. folly's libgoogle-glog-dev fix is not
            available for the next C++ repo.
   Fix: After each successful build, serialize the Dockerfile + language + package
        list + key lessons to a local SQLite/JSON store. On next repo of same
        language, embed top-3 most similar past Dockerfiles as few-shot examples.
   Cost: Moderate infrastructure work. Increases prompt size by ~500–1000 tokens
         per run. High long-term payoff at scale.

10. PROACTIVE SearchDockerError IN ANALYZE PHASE
    Problem: SearchDockerError is only called reactively after failures.
    Fix: Allow agent to call it during ANALYZE with questions like:
         "What are the most common pitfalls for building Gradle Java projects
         in Docker?" or "What is the correct base image for C++ with OpenSSL?"
    Cost: 1 extra SearchDockerError call per repo (~5–10s). ~2000 tokens.
    Benefit: Agents could discover known pitfalls before hitting them.

SECTION 8: KEY OBSERVATIONS FOR ACADEMIC PAPER
================================================================================

1. SUCCESS RATE
   68% overall (34/50). 62.8% among agent-generated repos (27/43).
   This is on a challenging, diverse set of production open-source repositories
   across 8+ languages and build systems.

2. FIRST-ATTEMPT EFFICIENCY
   63% of agent successes required only 1 attempt (17/27 agent-generated successes).
   If we had perfect language detection and base image validation, this likely
   increases to ~75–80%.

3. FAILURE MODE DISTRIBUTION
   The dominant failure mode (F-A: modify after verify) is a pure agent behavior
   bug that is entirely independent of repository complexity. It accounts for
   roughly 9 of 16 failures in some form.
   The second-dominant mode (Unknown language detection) correlates with 9 of 16
   failures and is a systematic preparation failure.
   Together these two fixable issues account for most of the failure budget.

4. TOKEN EFFICIENCY INVERSION
   Failed repos consumed 2.27× more tokens than successful ones, on average.
   This indicates the agent "tries harder" on hard repos but still cannot converge.
   For academic framing: the agent's sample efficiency degrades significantly for
   language=Unknown and complex C++/Java build systems.

5. PARALLELISM
   4 workers provided ~3.85× speedup. The near-linear scaling suggests that the
   bottleneck is LLM token generation + Docker build time, not Python overhead.
   There is no cross-repo state sharing, so parallel workers are fully independent.

6. LANGUAGE DIFFICULTY GRADIENT
   Python > Node.js > C (single binary) > C++ > Java (in descending success order).
   Java's difficulty is attributable to: Gradle/Maven plugin ecosystem complexity,
   Apache compliance tooling, and a tendency for language detection to fail on
   Java repos (which often have no obvious single entry point file).

7. ITERATIVE REFINEMENT VALUE
   The lesson-extraction + injection mechanism genuinely helps: 10 repos (37%) that
   needed more than 1 attempt still converged. The mechanism's Achilles' heel is
   the ordering violation bug, which corrupts the refinement loop rather than
   improving it.

8. NEAR-MISS ANALYSIS
   At least 3 repos (msgpack-c, webview, spring-framework) were within 1 correct
   attempt of succeeding. Increasing max_retries from 3 to 4 would likely recover
   these. The cost is ~30% more tokens for the failure population only.
